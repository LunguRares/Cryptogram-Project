We decided to follow the model of View> Controller> Model, and so that is reflected in the classes we have chosen.
The view is the user interface, which invokes the methods in the Controller. The Controller will call the appropriate methods in the model classes.
The model classes correspond to the each element of the game. The playerlist holds the players, and helps with the leaderboard generation.
The Player class holds all the information that corresponds with the player, i.e the statistics and the names. The CryptoGame class holds the game information, and the current gamestate.
